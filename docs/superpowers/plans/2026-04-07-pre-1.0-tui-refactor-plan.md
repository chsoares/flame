# Pre-1.0 TUI Refactor Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Refactor Flame into a safer TUI-first architecture before `1.0`, reducing correctness risks and removing legacy CLI-driven structural decisions without breaking the live app.

**Architecture:** This plan is intentionally phased around the riskiest seams first: session attribution, stdout-driven command execution, module runtime consolidation, and only then file reorganization. Each phase must be implemented and verified independently, with both automated tests and explicit manual validation in the real TUI app before proceeding.

**Tech Stack:** Go, Bubble Tea, Lip Gloss, existing internal session/runtime code, Go test suite, live TUI binary verification.

---

## Critical Workflow Rule

This plan has one non-negotiable workflow rule:

**Each phase must be completed, tested, and manually verified in the real app before moving to the next phase.**

That means:

- no batching multiple phases together
- no “we will test it later” behavior
- no spending days refactoring multiple subsystems before checking if the product still works

At the end of each phase, the implementing agent must:

1. run the focused automated tests for the changed area
2. run the broader relevant suite (`go test ./internal/...` at minimum unless the phase is extremely local)
3. build the app
4. tell the user exactly what changed
5. tell the user exactly what to test manually in the real app
6. stop and wait for confirmation before moving on

This is the main risk-control mechanism for the whole operation.

## Manual Verification Protocol For Every Phase

At the end of every phase, the implementing agent must provide a checkpoint message in this format:

```md
Phase X complete.

What changed:
- <1-3 bullets>

Automated verification:
- `go test ...`
- `go build -o flame .`

Please test in the real app before I continue:
1. `<exact command or interaction>`
2. `<exact command or interaction>`
3. `<exact command or interaction>`

Expected behavior:
- <what should happen>
- <what should not happen>
```

The manual test instructions must always be specific. Do not tell the user “test the app generally”. Always name the commands and the expected behavior.

## Scope Boundary

This plan assumes:

- TUI is the shipping product
- CLI-only behavior is legacy and should not drive architecture decisions
- code that exists only to support the old standalone CLI can be removed, bypassed, or collapsed if that improves the TUI architecture

This plan does **not** attempt a full codebase cleanup. It focuses on the architectural work that should happen before `1.0` and prepares the codebase for the later coverage/dead-code/consistency pass.

Add tests that cover at least:

- worker pending state is cleared on failure paths
- worker pending state is not left stuck after timeout

- only the intended connection path is marked special

At minimum, add targeted tests around the current stateful helpers rather than trying to simulate the full network stack immediately.

- [ ] **Step 2: Run the focused tests and verify they fail for the expected reason**


- no stale `pendingWorker`
- no stale `pendingSSH`
- all early returns clear pending state correctly
- timeout/failure paths clear pending state correctly
- if a dedicated request object can be introduced safely in this phase, do so; otherwise make the current state machine safe first

Do **not** bundle this with full file splitting yet.

- [ ] **Step 4: Re-run the focused attribution tests**

Run:


Start with synchronous command output paths that are easiest to make explicit, such as:

- `help`
- `modules`
- `config`
- `sessions`

Do not try to convert uploads/downloads/modules/spawn in the same sub-step.

- [ ] **Step 2: Write failing tests for explicit command output return values**

Add or update tests so the chosen commands are exercised through the TUI-facing execution path without relying on stdout capture.

- [ ] **Step 3: Run the focused tests and verify they fail correctly**

```bash

```

- [ ] **Step 4: Implement explicit result-returning command paths for the chosen command slice**

The key outcome is:

- TUI-facing execution no longer needs stdout capture for these commands
- the manager layer returns text/results explicitly

This phase may leave some legacy print-driven code for more complex paths temporarily, but the shipped TUI path must stop depending on process-global output mutation for the converted commands.

- [ ] **Step 5: Expand to the remaining synchronous command slice if still small enough**

Only continue in this phase if the next command slice is similarly safe and can still be verified independently. If not, stop this phase after the first successful slice and commit it as an incremental checkpoint.

## Relationship To The Future Coverage / Audit Plan

This plan intentionally overlaps with the future coverage/audit effort only where the overlap is useful.

Use this rule:

- if a test directly protects a seam being refactored now, add it now
- if a cleanup is obvious and safely local to the changed area, do it now
- if a dead-code/coverage/style issue is unrelated to the active phase, defer it to the broader later audit

That keeps the current refactor disciplined and prevents it from turning into an open-ended whole-repo cleanup.
