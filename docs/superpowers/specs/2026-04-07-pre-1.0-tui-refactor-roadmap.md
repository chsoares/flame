# Pre-1.0 TUI Refactor Roadmap

Date: 2026-04-07

## Purpose

This document defines the refactor roadmap that should guide the final architecture cleanup before `1.0`, with a strict TUI-first assumption.

It is written as a handoff document for a future agent or engineer who may not have the context of earlier conversations.

## Product Direction

Flame will be launched as a **TUI application**.

The old CLI-oriented behavior is legacy code from the first version. It should no longer drive architecture decisions. Code that exists only to support the standalone CLI should be treated as expendable. That does not mean it must all be deleted immediately, but it should not be preserved at the cost of cleaner TUI architecture.

This is the key lens for every decision below:

- TUI is the product.
- Shared backend logic should exist to serve the TUI.
- CLI-only adapters, output-capture tricks, and file organization created only to preserve both modes can be simplified, replaced, or removed.

## Current Situation

The codebase was not built uniformly from the beginning. Early parts of the project were developed without TDD and without structured code review. Later work introduced better discipline, tests, and more deliberate file boundaries.

As a result, the current codebase has a mixed character:

- some newer areas are already reasonably modular
- some older areas still have god files and mixed responsibilities
- some execution flows diverge between CLI and TUI
- some infrastructure still relies on global stdout capture and print-driven control flow

This matters because the project is approaching release, and release architecture should be shaped around the product that actually ships.

## Important Constraint

There is already a separate future plan for broad testing cleanup, coverage audit, dead code removal, and consistency work. That future plan is still valuable, but this roadmap should prioritize the work that is most useful while major architectural areas are already being touched.

In other words:

- not everything from the future audit plan must happen in this step
- but this refactor step should create cleaner boundaries so the later test/coverage pass becomes much easier and safer

## Existing Strengths

Not everything needs restructuring.

Areas that already look directionally correct:

- `internal/tui/modal.go` + `internal/tui/dialog.go` + `internal/tui/helpmodal.go`
  - Shared shell + modal-specific content is a good pattern.
- `internal/tui/layout.go`
  - Focused and coherent.
- `internal/tui/outputpane.go`
  - Large, but the responsibilities are related enough to justify the file.
- `internal/help.go`
  - Central help data is the right shape for reuse across terminal and modal views.
- `internal/modules.go`
  - The thin module definitions are readable; the bigger issue is the execution plumbing around them, not the individual module wrappers.

This roadmap is not about rewriting everything. It is about fixing the places where the current architecture will actively get in the way of release quality and post-release maintenance.

## Main Architectural Problems

### 1. `internal/session.go` is a god file

This file currently mixes several unrelated responsibilities:

- session lifecycle
- shell relay plumbing
- session detection
- module execution
- transfer execution
- command routing
- completion logic
- TUI-facing executor methods
- output capture and print-driven command behavior

This is a problem because:

- changes in one subsystem increase risk in unrelated ones
- tests become harder to write because setup drags in too much state
- duplication survives longer because there is no obvious home for shared orchestration helpers
- file size makes review and safe edits harder

This is the single biggest code organization problem in the repo.

### 2. `internal/tui/app.go` is also too broad

`app.go` currently mixes:

- Bubble Tea update loop
- context switching between menu/shell/overlays
- command routing
- shell echo/prompt logic
- menu output append logic
- splash rendering
- sidebar rendering
- view composition
- runtime/wiring behavior

This is less dangerous than `session.go`, but it is still a major maintainability problem.

### 3. Module execution architecture is split between CLI and TUI

The current `run` behavior is not unified:

- CLI path executes differently
- TUI path uses worker shell orchestration

This creates drift in semantics, alias behavior, cancellation, and error handling.

Since Flame is TUI-first, the module runtime should be designed once around the TUI execution model, and legacy CLI behavior should stop defining the architecture.

### 4. Session attribution for workers/SSH is fragile

Current worker/SSH classification relies on global pending flags and “next session wins” logic.

That is a real correctness problem, not merely a style issue.

Potential failure modes include:

- unrelated incoming session classified as worker
- stale pending state after timeout/error
- concurrency hazards when multiple asynchronous flows overlap

This is exactly the type of bug that can hide until a stressful real-world run.

### 5. Print-driven command execution still shapes the internals

Some backend command paths still effectively use stdout as an API.

This leads to:

- global stdout capture
- awkward TUI integration
- difficult testing
- concurrency risk

For a TUI-first product, command execution should return data or structured text directly to the TUI layer instead of pretending stdout is a stable control boundary.

## Refactor Strategy

The refactor should not be treated as one giant cleanup. It should be staged in a careful order that reduces risk and unlocks later work.

The best order is:

1. fix correctness risks that can cause misbehavior
2. replace dangerous architectural seams
3. split files along responsibility boundaries
4. only then do broader cleanup, test expansion, and style consistency work

This ordering matters. If file splitting happens before the dangerous seams are understood, it is easy to scatter bad patterns across more files instead of actually improving the design.

## Must-Do Before `1.0`

These items are recommended as the pre-`1.0` refactor scope.

### Must-Do 1: Replace fragile pending worker / pending SSH attribution

**Problem**

Worker and SSH session classification currently relies on shared global state that marks the “next” incoming session as special.

This is fragile because the system is asynchronous. Any failure path, timeout, or unrelated incoming shell can break that assumption.

**Why this is pre-1.0 material**

This is not cosmetic. It is a correctness and stability risk in session handling, which is core product behavior.

**Proposed solution**

Replace global pending booleans with a real correlation mechanism.

Preferred direction:

- introduce a pending session request object or coordinator
- create the request only after payload dispatch succeeds
- always clear it on timeout/failure
- ideally correlate using an explicit token rather than “newest session” or “next incoming session”

If token-based correlation is too large for this stage, the minimal acceptable pre-`1.0` fix is:

- no stale pending flags
- no unbounded “next session” classification
- clear ownership of pending worker lifecycle

**Why this should happen first**

Because later refactors to module runtime and session organization should sit on top of safer session attribution.

**Testing value**

This work will directly support the later coverage/testability plan because it forces clearer boundaries around session request lifecycle.

### Must-Do 2: Remove stdout-capture architecture from TUI-facing command execution

**Problem**

Some command paths still rely on `fmt.Println` plus stdout capture/redirection. That is effectively using process-global IO as an internal API.

This is especially risky in a concurrent TUI app.

**Why this is pre-1.0 material**

It is both an architectural smell and a practical source of race-prone behavior.

**Proposed solution**

Move command execution toward explicit return values or structured results.

Preferred direction:

- command handlers return text/results instead of printing
- TUI decides how to append/render them
- background flows use explicit sinks/callbacks instead of temporarily mutating `os.Stdout`

This does not require rewriting every code path at once. The key pre-`1.0` step is to remove the architectural dependence of the shipped TUI on stdout capture.

**Why this should happen before major file splitting**

Because if you split files first, you will simply redistribute the print-driven design into more files. Better to replace the seam first and then reorganize the code around the cleaner seam.

**Connection to later audit plan**

This strongly supports Task 2 and Task 3 from the future testability plan. Once commands return values instead of printing, writing tests becomes dramatically easier.

### Must-Do 3: Consolidate module execution into one TUI-first runtime path

**Problem**

The code currently has divergent execution logic for `run`, depending on path and context.

That divergence creates inconsistencies in:

- worker behavior
- alias handling
- cancellation
- user expectations
- help text accuracy

**Why this is pre-1.0 material**

Module execution is a central feature. This should not remain split-brain at release time.

**Proposed solution**

Introduce one module execution runtime/service used by the TUI.

The core flow should become:

1. resolve module name and aliases
2. validate session selection and execution prerequisites
3. choose execution target/runtime policy
4. spawn worker if required by the TUI execution model
5. execute module with shared orchestration code
6. report progress/errors through TUI-friendly channels

The important architectural change is that the runtime becomes authoritative and TUI-oriented. Legacy CLI behavior should not define the flow anymore.

**Scope guidance**

Do not overbuild a giant framework here. The goal is one coherent runtime path, not an abstract plugin platform.

### Must-Do 4: Break up `internal/session.go`

**Problem**

Even if behavior improves, the current file is still too large and too mixed to be a healthy long-term home for the session subsystem.

**Why this is pre-1.0 material**

Because the shipped product will continue to evolve in this area, and leaving it as one huge file makes all post-release maintenance riskier.

**Proposed file split**

Recommended minimum split:

- `internal/session_manager.go`
  - session state, add/remove/select, relay lifecycle, session queries
- `internal/session_detection.go`
  - platform/whoami/shell detection
- `internal/session_commands.go`
  - command routing and command handlers that still belong to manager-level concerns
- `internal/module_runtime.go`
  - worker orchestration and shared module execution flow
- `internal/session_completion.go`
  - autocomplete/completion logic
- `internal/tui_executor.go`
  - the specific methods the TUI uses from the manager-facing interface

This split is by responsibility, not by technical layer.

**What not to do**

- do not split it into dozens of tiny files
- do not preserve CLI/TUI duality as the organizing principle
- do not move code mechanically without improving seams first

### Must-Do 5: Break up `internal/tui/app.go`

**Problem**

`app.go` is doing too much and already has duplicated transition logic.

**Why this is pre-1.0 material**

Because the TUI is the shipped product. Its top-level model should be maintainable before release.

**Proposed file split**

Recommended minimum split:

- `internal/tui/app_update.go`
  - top-level update flow and message routing
- `internal/tui/app_commands.go`
  - menu command dispatch, bang command dispatch, command helper methods
- `internal/tui/app_shell.go`
  - prompt detection, local echo rules, shell-specific helpers
- `internal/tui/app_view.go`
  - `View`, splash/sidebar rendering, line padding, view assembly
- `internal/tui/app_runtime.go`
  - executor wiring, startup/runtime helpers that are not model logic

**Additional cleanup required during this split**

Create orchestrator helpers for repeated transitions, such as:

- setting app context
- entering shell mode
- leaving shell mode
- recomputing/applying layout
- executing a command and appending output consistently

This is exactly the kind of improvement that prevents future drift.

### Must-Do 6: Centralize alias resolution and execution preflight helpers

**Problem**

Some command normalization and preflight checks are duplicated or split across layers.

Examples include:

- module alias normalization
- selected-session-required checks
- output append + newline + sync behavior
- layout application on resize/collapse

**Why this is pre-1.0 material**

These are not huge features individually, but they create drift and inconsistency quickly.

**Proposed solution**

As part of the `session.go` and `app.go` cleanup, extract small parent helpers that own shared orchestration.

Examples:

- canonical module resolution helper in one place
- selected session guard helper
- menu command execution helper
- layout apply helper
- shell context transition helper

The modal refactor already proved this pattern is worth doing: one shell/template function removed duplicated behavior and made future changes safer.

## Nice To Have After `1.0`

These are worthwhile, but they are lower priority than the must-do items above.

### Nice-To-Have 1: Broader dead code and unused seam cleanup

Examples to revisit after `1.0`:

- unused messages in the TUI message model
- stale focus modes or partial feature seams
- helpers only used by old CLI-era behavior
- tests that codify abandoned execution assumptions

This aligns well with the future audit plan’s dead code and redundancy step.

### Nice-To-Have 2: Broader module organization cleanup

`internal/modules.go` can eventually be split into:

- `module_registry.go`
- `modules_linux.go`
- `modules_windows.go`
- `modules_custom.go`

That said, this should happen after the module runtime is stabilized. Splitting wrappers before fixing runtime would mostly be cosmetic.

### Nice-To-Have 3: Refactor duplicate runner internals further

There are likely reusable helpers around:

- source resolution
- output file setup
- delivery strategy
- remote command construction

That refactor becomes more valuable once the runtime path is unified and safer.

### Nice-To-Have 4: Shared rendering helpers for small UI patterns

Some repeated status/footer/hint rendering patterns can be extracted later.

This is useful, but not a release blocker.

### Nice-To-Have 5: Full CLI removal

Once the TUI-first architecture is in place and release is out, the codebase can be more aggressively cleaned of remaining CLI-only assumptions and paths.

This is likely the right post-`1.0` simplification pass.

## Recommended Execution Order

The order below is intended to minimize risk and avoid refactoring into the wrong abstractions.

### Phase 1: correctness and seam replacement

1. Replace fragile worker/SSH pending attribution.
2. Remove stdout-capture dependence from shipped TUI command execution.
3. Centralize module alias resolution.

These changes establish safer behavior and cleaner boundaries.

### Phase 2: runtime consolidation

4. Unify module execution into one TUI-first runtime path.
5. Clean up related session/runtime helpers around worker execution.

This avoids preserving the split CLI/TUI mental model.

### Phase 3: file boundaries

6. Split `internal/session.go` by responsibility.
7. Split `internal/tui/app.go` by responsibility.

At this stage, file splitting is structural clarification, not just moving clutter around.

### Phase 4: targeted cleanup and tests

8. Add/refine tests for newly-isolated seams.
9. Remove stale helpers/types that became unnecessary after the refactor.

This is where the future testing and audit plan starts to become much cheaper and safer.

## How This Interacts With the Future Testing / Coverage Plan

The future plan described elsewhere remains valuable, especially for:

- coverage audit
- missing tests
- dead code review
- standards/consistency cleanup

However, the order should be interpreted carefully.

### What should happen now during this refactor step

- add tests for the new seams introduced by the refactor
- use minimal structural refactors to improve testability where needed
- remove obviously dead or redundant code created obsolete by the new architecture

### What can wait for the later audit step

- broad coverage improvement across the whole repository
- lower-risk consistency cleanup
- dead code review in areas untouched by the architectural refactor
- style-level cleanup that is not unlocking release confidence

### Why this order is better

If broad coverage work happens before the architectural cleanup, there is a real risk of writing tests against the wrong seams, then paying for churn during refactor.

If the architecture is improved first in the riskiest areas, the later test/coverage pass can target clearer responsibilities with more stable boundaries.

## Explicit Non-Goals For This Step

To keep scope disciplined before `1.0`, this roadmap does **not** recommend:

- rewriting the entire codebase
- abstracting every repeated line into a helper
- designing a generic framework for modules or commands
- preserving clean coexistence between legacy CLI and TUI
- chasing 100% coverage

The goal is not theoretical neatness. The goal is a safer, cleaner release architecture for the product that will actually ship.

## Decision Summary

Before `1.0`, prioritize the architecture changes that:

- reduce real behavioral risk
- remove dangerous seams inherited from CLI-era design
- unify core runtime behavior around the TUI
- split the largest god files only after the seams are improved

After `1.0`, use the testing/audit plan to widen coverage, remove more dead code, and continue cleanup from a healthier base.

## Short Version For The Next Agent

If you only remember one thing, remember this:

**Do not preserve CLI legacy at the expense of TUI architecture.**

The most important pre-`1.0` work is:

1. fix fragile worker/SSH session attribution
2. stop using stdout capture as a control boundary
3. unify module execution into one TUI-first runtime path
4. then split `session.go` and `tui/app.go` along real responsibilities

Everything else is secondary.
